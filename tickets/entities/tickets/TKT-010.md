## Description

Add support for validating markdown content structure within entity bodies, enabling required headers validation (e.g., ADR sections) with exact and regex matching.

## Summary

Add support for validating markdown content structure within entity bodies. This enables
enforcing required headers (e.g., ADR sections), checking for prohibited patterns, and
ensuring content completeness based on entity status.

## Design

### Syntax

Extend validation rules with a `content` section:

```yaml
validations:
  - name: done-tickets-need-resolution-section
    entity_type: ticket
    when:
      - "status=done"
    content:
      required-headers:
        - "## Description"
        - "## Resolution"
    severity: error

  - name: decisions-need-adr-sections
    entity_type: decision
    when:
      - "status=accepted"
    content:
      required-headers:
        - "## Context"
        - "## Decision"
        - "## Consequences"
        - pattern: "## (Alternative|Alternatives)"  # regex
    severity: error
```

### Header Matching

- **String** (e.g., `"## Context"`) - exact match
- **Map with `pattern` key** (e.g., `pattern: "## (Alternative|Alternatives)"`) - regex match

### Future Extensions (not in initial scope)

| Check | Description |
|-------|-------------|
| `ordered: true` | Headers must appear in specified order |
| `non-empty-sections: true` | Headers must have content beneath them |
| `no-patterns: [...]` | Ban patterns like TODO/FIXME |
| `has-patterns: [...]` | Require patterns like checkboxes |
| `min-content-length: N` | Minimum body length |

## Implementation

1. Add `ContentRule` and `HeaderCheck` types to `internal/metamodel/types.go`
2. Extend `ValidationRule` with optional `Content *ContentRule` field
3. Add content validation logic in `internal/dataentry/analyze.go`
4. Parse markdown headers from `entity.Content` field
5. Match against required headers (exact or regex)
6. Report violations through existing `analyze_validations` flow

## Acceptance Criteria

- [ ] Can define `required-headers` in validation rules
- [ ] Exact string matching works
- [ ] Regex matching via `pattern:` works
- [ ] Violations appear in `analyze_validations` output
- [ ] Works with `when:` conditions (status-based requirements)
- [ ] Error vs warning severity respected
