## Problem\n\nAfter TKT-PYN1 lands, every handler reads state via `atomic.Pointer.Load()`. But `App.mu` is still held as RWMutex because mutation handlers need to serialize writes against each other and against reloads. The current pattern is a lock-upgrade dance (`RUnlock → Lock → defer(Unlock → RLock)`) that is fragile and obscures intent.\n\nSolution: split the two responsibilities. Introduce a dedicated `writeMu sync.Mutex` that mutation handlers take around the mutation call. `App.mu` stays as RWMutex for reload-vs-write coordination, but the upgrade dance goes away because handlers no longer need exclusive access to the *state pointer* — only to the *write path*.\n\nSee `.ignored/locking-alternatives.md` §5 Stage 3 for the full plan.\n\n## Depends On\n\n- TKT-PYN1 (AppState refactor)\n\n## Scope\n\n**In scope:**\n\n- Add `writeMu sync.Mutex` to `App`.\n- In every mutation handler (`handleV1CreateEntity`, `UpdateEntity`, `DeleteEntity`, `CreateRelation`, `UpdateRelation`, `DeleteRelation`, `SetProperty`, `handleV1Action`), replace the lock-upgrade dance with `a.writeMu.Lock() / defer a.writeMu.Unlock()` around the mutation call.\n- Keep `App.mu.RLock` from `reloadLockMiddleware` — still needed to prevent reloads mid-request.\n- Verify that concurrent mutation handlers are still properly serialized (same semantics as before).\n- Tests that manually simulated the middleware (holding `app.mu.RLock` in test setup) may need updating.\n\n**Out of scope:**\n\n- Deleting `App.mu` or `reloadLockMiddleware` (ticket T4).\n- Narrowing the script-contract invariant (separate future ticket, Stages 5-6).\n- Any behavior change visible to clients.\n\n## Acceptance Criteria\n\n1. The lock-upgrade dance (`RUnlock → Lock → defer(Unlock → RLock)`) is gone from every mutation handler.\n2. Mutation handlers acquire `writeMu` around the mutation call.\n3. Two concurrent mutation requests still serialize correctly (test with `-race`).\n4. Reload during a mutation still works (test covers this).\n5. All tests pass under `-race`.\n\n## Risk\n\nThis stage is a subtle invariant shift: before, the mutation handler held `App.mu.Lock` which excluded reloads. After, the mutation handler holds `writeMu` (which excludes other mutations) AND `App.mu.RLock` (which excludes reloads — reload needs `App.mu.Lock`). So reload serialization is preserved via a different path. This needs a clear test that exercises reload-during-mutation.
