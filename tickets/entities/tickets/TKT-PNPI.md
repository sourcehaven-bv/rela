## Problem\n\nThe workspace today has no unified mutation primitive. Every writer (`CreateEntity`, `UpdateEntity`, `DeleteEntity`, `CreateRelation`, `DeleteRelation`, `rename.Rename`, `importer`) hand-rolls its own sequence of: validate, persist via `repo.Transaction`, then mutate the in-memory graph and search index. The locking discipline is encoded at every call site instead of being an invariant of the workspace. As a result:\n\n- `rename` is ~280 lines of orchestration that could be a few dozen lines of intent.\n- Automation cascades re-enter the workspace via `w.scriptExec.ExecuteCode(ctx)` which then calls `workspace.CreateEntity` recursively — there is no shared notion of "this is one transaction". A half-applied cascade cannot be rolled back.\n- The fix for the graph-torn-during-reload race (originally scoped as this ticket) and the eventual move to real snapshot isolation both want the same primitive: a `Tx` that stages changes against a base snapshot and commits them atomically.\n\nDiscovered while scoping the deferred critical findings from TKT-252Y's cranky review (RR-PA4Y, RR-FJH7). Rather than fix the narrow graph-swap problem in isolation, introduce the transactional primitive that makes the fix (and several future improvements) fall out naturally.\n\n## Framing\n\nOnly `meta` is primary state. Everything else in the workspace — `graph`, `searchIdx`, `automation` — is a derived view over `meta` plus the persisted entities and relations on disk. The in-memory graph is a cache of disk, not a source of truth. Under this framing:\n\n- A `State` is an immutable bundle of `{meta, graph, searchIdx, automation}`, held via `atomic.Pointer[State]`.\n- Readers `Load()` once and work against a coherent snapshot.\n- Writers do not mutate a `State`. They describe their mutation as a `Tx` and `Commit()` produces a new `State` containing the post-mutation graph and index.\n- Initially, `Commit` mutates the base `State`'s graph and search index **in place** under `writeMu`, because cloning on every CRUD would be expensive and rela has no current need for snapshot isolation across CRUD operations. This is "transactional in name and API, mutable under the hood."\n- A future ticket can upgrade `Commit` to clone-and-publish without touching any caller, once the `Tx` API is in place.\n\n## Scope\n\n**In scope:**\n\n1. Introduce `workspace.State` struct (renamed from the current internal `workspaceState`) as the public-ish snapshot type with `meta/graph/searchIdx/automation` fields. Still unexported; accessed via `ws.State()` accessor.\n2. Introduce `workspace.Tx` type:\n   - Holds a reference to the base `State` and the `Workspace`.\n   - Read methods: `GetEntity`, `GetRelation`, `IncomingEdges`, `OutgoingEdges`, `AllEntities`, etc. — each reads from the base snapshot (for now; once staging is layered, reads go through a staging view).\n   - Write methods: `Create`, `Update`, `Delete`, `CreateRelation`, `UpdateRelation`, `DeleteRelation`. Initially these apply **immediately** to the base snapshot's mutable graph — no staging buffer yet. The abstraction is in place; the lazy-staging upgrade is a follow-up.\n3. Introduce `ws.WithTx(fn func(tx *Tx) error) error`:\n   - Acquires `writeMu`.\n   - Loads the current `State` as the base.\n   - Opens a single `repo.Transaction` for disk writes.\n   - Runs the closure; closure mutations go through `tx.Create/...` which persist to disk via the repo tx and mutate the graph in place.\n   - Commits the repo tx on success; rolls back on error.\n   - Releases `writeMu`.\n4. Reimplement `Workspace.CreateEntity`, `UpdateEntity`, `DeleteEntity`, `CreateRelation`, `UpdateRelation`, `DeleteRelation`, `SetProperty` as thin wrappers around `WithTx`. External API (signatures, return types, error semantics) is preserved; only the internal plumbing changes.\n5. Migrate `rename.Rename` to use `ws.WithTx` — the function shrinks to ~50 lines of intent. Its current `executeRename` and `updateGraph` helpers are deleted.\n6. Migrate `importer.Import` to use `ws.WithTx` for bulk entity/relation creation. Bulk imports become one large transaction instead of many small ones.\n7. Change `repo.Sync` to return a fresh graph instead of mutating one in place. `Reload` publishes a new `State` with `newGraph, newIdx, newAuto` atomically. This closes the graph-torn-during-reload issue from RR-PA4Y/RR-FJH7 because reload no longer mutates the current graph — it builds a new one.\n8. Strengthen `TestConcurrentReloadStateSnapshot` to include readers that iterate the graph (e.g. via `CheckCardinality`), removing the "pre-existing limitation" caveat.\n9. Delete `graph.Clear()` from `internal/graph/graph.go` if no callers remain (it was only used by `repo.Sync`).\n10. Update the `Workspace` struct doc comment to remove the "Known limitation: graph-torn-during-reload" section.\n\n**Out of scope (separate follow-up tickets):**\n\n- **Routing automation re-entrance through the Tx** (TKT-AUTX, filed as a follow-up). Today, an automation's Lua script calls `rela.create_entity` which re-enters the workspace via `scriptContextImpl` → `workspace.CreateEntity` → a **new, nested** `WithTx`. This works correctly as long as the nested `WithTx` detects that the current goroutine already holds `writeMu` and reuses the open repo transaction instead of opening a nested one. The minimal version of this ticket will implement nested-tx-reuse via a goroutine-local or context-carried "current tx" pointer. The **proper fix** — making automation stage changes against the outer tx so they commit atomically — is a follow-up.\n- **Clone-and-publish commit semantics** (TKT-ISOC, filed as a follow-up). Once Tx is in place, a future ticket can change `commit` to clone the base graph and search index, apply staged changes to the clones, and publish a new `State`. This gives real snapshot isolation across CRUD operations. Deferred because rela has no current reader that needs it and the clone cost is real.\n- **Persistent data structures for the graph** (HAMTs). Long-term option, probably never needed.\n- **True staging buffer inside `Tx`** where writes accumulate and are applied at commit time. Initial version applies mutations immediately. The staging upgrade is a follow-up and becomes valuable only when clone-and-publish lands.\n\n## Acceptance Criteria\n\n1. `workspace.Tx` type exists with read and write methods, and `Workspace.WithTx(fn)` runs a closure inside a tx.\n2. All existing `Workspace.CreateEntity/UpdateEntity/DeleteEntity/CreateRelation/UpdateRelation/DeleteRelation/SetProperty` calls internally go through `WithTx`. External signatures and semantics are unchanged.\n3. `rename.Rename` is rewritten to use `ws.WithTx`. The `executeRename` and `updateGraph` helpers are deleted. The function is materially shorter than today.\n4. `importer.Import` uses `ws.WithTx` for bulk operations.\n5. `repo.Sync` returns a fresh `*graph.Graph` instead of mutating one in place. `graph.Clear` is deleted if no callers remain.\n6. `Workspace.Reload` publishes a new `State` containing a freshly-built graph, search index, and automation engine — no in-place mutation of the previous graph.\n7. `TestConcurrentReloadStateSnapshot` is extended to iterate the graph in reader goroutines and passes under `-race`.\n8. The "Known limitation: graph-torn-during-reload" section is removed from the `Workspace` struct doc comment.\n9. All existing tests pass under `-race`. No behavior change observable to callers other than `rename` and `importer` (which are implementation details).\n10. `go test -race ./...`, `just lint`, and `just coverage-check` all pass.\n\n## Dependencies\n\n- TKT-252Y (done): established the `atomic.Pointer[workspaceState]` pattern that this ticket builds on.\n\n## Risk\n\n- **Nested transactions from automation.** This is the subtlest part. The minimal version reuses an outer tx when a nested call is detected (same goroutine, writeMu already held, repo tx already open). Tests must cover: single-level write, write-triggers-automation, automation-creates-entity-triggers-further-automation. The follow-up ticket TKT-AUTX will replace this minimal version with proper tx-aware automation staging.\n- **`repo.Sync` signature change** ripples into ~5 callers (New, Reload, CLI sync command, two tests). Mechanical but touches package boundaries.\n- **Mass migration of existing workspace mutation methods to WithTx** touches every test that mocks or exercises mutation. Most should be green with no changes because the external signatures are preserved. A small number may need fixture updates.\n\n## Why this supersedes the original TKT-PNPI scope\n\nThe original scope was "eliminate graph-torn-during-reload race" — a narrow fix to make `repo.Sync` return a new graph. That fix is still in scope (AC5/6), but it's now part of a coherent larger change that also:\n\n1. Gives us a unified mutation primitive (`Tx`).\n2. Sets up the infrastructure for future snapshot isolation without requiring another migration.\n3. Collapses `rename`'s custom orchestration into shared code.\n4. Lays the groundwork for atomic automation cascades.\n\nThe narrow fix alone would have closed RR-PA4Y/RR-FJH7 but left the workspace with the same "every writer hand-rolls its own locking discipline" problem. Doing the framing step now means future writers (flows, MCP tools, etc.) automatically get the transactional semantics for free.
