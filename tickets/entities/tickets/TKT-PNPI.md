## Problem\n\nThe workspace today has no unified mutation primitive. `rename.Rename` is ~280 lines of orchestration that hand-rolls validate → `repo.Transaction` → mutate-graph-in-memory. Other writers (`CreateEntity`, `UpdateEntity`, etc.) don't even use `repo.Transaction` — they write to disk immediately and update the graph after. There's no abstraction that bundles disk persistence with deferred graph mutation under a single rollback boundary.\n\n## Already shipped (do NOT redo)\n\nDelivered by TKT-Z7HL (PR #342):\n- ✅ `repo.Sync` returns a fresh graph\n- ✅ `Workspace.Reload` publishes new state atomically\n- ✅ `graph.Clear()` deleted\n- ✅ `TestConcurrentReloadStateSnapshot` iterates the graph in readers\n\nDelivered by TKT-PYN1/WYYP/9NFK (PR #347):\n- ✅ `App.writeMu` serializes data-entry mutation handlers\n- ✅ `App.mu` and `reloadLockMiddleware` deleted\n\n## Scope (this PR — minimum viable Tx)\n\n1. Introduce `workspace.Tx` type in `internal/workspace/tx.go`. Holds:\n   - Reference to the `Workspace`\n   - Base `workspaceState` snapshot loaded at tx entry (read methods see this)\n   - Open `repository.Tx` from `repo.Transaction`\n   - Pending graph mutations (addNodes/removeNodes/addEdges/removeEdges) applied only after disk commit succeeds\n\n2. Tx read methods (delegate to base snapshot): `Meta`, `Graph`, `GetEntity`, `GetRelation`, `IncomingEdges`, `OutgoingEdges`.\n\n3. Tx write methods (stage to repo tx + record graph op): `WriteEntity`, `WriteRelation`, `DeleteEntity`, `DeleteRelation`.\n\n4. `Workspace.WithTx(fn func(tx *Tx) error) error`:\n   - Acquires `reloadMu` (existing internal mutex). This excludes concurrent Reload, Sync, other WithTx calls.\n   - Loads the current snapshot.\n   - Calls `repo.Transaction(repoTx => fn(workspaceTx))` — disk operations stage to .new files; on closure success they atomically rename, on closure error they roll back.\n   - On commit success: applies pending graph mutations, saves cache.\n   - On rollback: graph is untouched, no cache save.\n\n5. New tests in `internal/workspace/tx_test.go`:\n   - `TestWithTxCommit`: closure does WriteEntity. After WithTx returns nil, the entity is on disk AND in the graph.\n   - `TestWithTxRollback`: closure does WriteEntity then returns an error. After WithTx returns the error, the entity is NOT on disk and NOT in the graph (rollback verified).\n   - `TestWithTxMixedOps`: closure does WriteEntity + WriteRelation + DeleteEntity. All succeed atomically.\n\n**Out of scope (follow-up tickets):**\n\n- **Migrating `rename.Rename` to use `WithTx`.** Requires either an import cycle workaround or moving 280 lines into the workspace package. Worth doing as its own focused PR.\n- **Migrating `CreateEntity`/`UpdateEntity`/`DeleteEntity`/`CreateRelation`/etc.** Behavior change (automation cascades become atomic) needs its own focused PR.\n- **Nested-tx detection** for automation reentrance. Not needed yet — no caller in this PR triggers it.\n- **`importer.Import` migration.**\n- **TKT-CXHM (automation staging on outer Tx).**\n- **TKT-1LFQ (clone-and-publish commit semantics).**\n\n## Acceptance Criteria\n\n1. `workspace.Tx` type exists with all listed read + write methods.\n2. `Workspace.WithTx(fn)` exists and uses `repo.Transaction` internally.\n3. New `TestWithTxCommit`, `TestWithTxRollback`, `TestWithTxMixedOps` tests pass.\n4. All existing tests pass under `-race`.\n5. `go test -race ./...`, `just lint`, `just coverage-check` all pass.\n6. The Tx primitive doc comment explains the deferred-graph-mutation model and the rollback semantics.\n\n## Effort\n\n**xs** — the Tx code is already written (~150 lines of tx.go + WithTx in workspace.go). Just need the tests + lint + review. ~30 minutes.
