## Problem\n\nAfter TKT-PNPI lands, `Workspace.WithTx` commits by mutating the base `State`'s graph and search index **in place** under `writeMu`. This matches today's semantics and is correct for rela's current workload, but it means a reader that `Load()`s a `State` before a commit can observe post-commit effects on the "pre-commit snapshot" (because the graph pointer inside their snapshot is the same pointer we mutate).\n\nIn other words: the `State` is reference-immutable but its `graph` field is mutable. True per-operation snapshot isolation requires a reader's snapshot to remain byte-stable for the reader's lifetime, regardless of concurrent commits.\n\nThis ticket upgrades `Commit` to **clone the graph, apply changes to the clone, and publish a new State**. No caller changes.\n\n## Depends on\n\n- TKT-PNPI — provides the `Tx` abstraction that makes this a single-function change.\n\n## Scope\n\n**In scope:**\n\n- Add `Graph.Clone()` returning a deep-copied `*graph.Graph` with the same nodes, edges, adjacency maps, and property index.\n- Update `Tx.commit()` to: clone the base graph, apply all staged changes to the clone, publish a new `State` containing the clone.\n- For the search index: since bleve does not support cheap cloning, either (a) rebuild the index from the new graph on every commit (expensive — O(n) per commit; probably unacceptable), or (b) accept that the search index lags committed graph state briefly and is eventually rebuilt on the next `Reload`. Pick (b) initially and document.\n- Remove the `writeMu` held-across-commit requirement once clone-and-publish lands, if safe. (It may still be needed to serialize committers against each other so they don't both clone the same base and race on publication; keep it to be safe.)\n- Strengthen concurrency tests: a reader that loads the state before a commit should observe the pre-commit graph even after the commit completes.\n\n**Out of scope:**\n\n- Cheap search index cloning. If bleve ever supports it, revisit.\n- Persistent data structures for the graph.\n- Staging buffer inside Tx (the tx still applies to the clone immediately during Commit; the staging upgrade is a separate improvement).\n\n## Acceptance Criteria\n\n1. `Graph.Clone()` exists and produces an independent deep copy.\n2. `Tx.commit()` uses clone-and-publish semantics for the graph.\n3. A new test verifies that a reader which loads `ws.State()` before `Commit` observes the pre-commit graph after commit completes.\n4. Search index semantics are explicitly documented (it lags commits; the next reload rebuilds it).\n5. No performance regression visible in the existing benchmark suite.\n6. `go test -race ./...`, `just lint`, `just coverage-check` all pass.\n\n## Trade-offs (for review during planning)\n\n**Pros:**\n- True per-operation snapshot isolation for graph reads.\n- Enables future features that need stable multi-query snapshots (long-running analyze, export, flows that collect data across steps).\n- Closes the last caveat in the Workspace concurrency model.\n\n**Cons:**\n- Every commit pays `O(|nodes| + |edges|)` to clone. For a project with 5000 entities and 20000 edges, this is ~hundreds of microseconds plus GC pressure.\n- Automation cascades that create 50 entities in one tx still clone once at commit — that's fine — but multiple small commits become more expensive than today.\n- Search index divergence window is user-visible (search results lag committed data for some interval).\n\n**Recommendation during planning:** benchmark the clone cost on a realistic graph before committing to this. If clone time is noise, ship it. If it shows up in the hot path, consider persistent data structures or leave clone-and-publish as opt-in via a new API.
