## Problem\n\nThe interactive Lua flow system (FEAT-YIOF, TKT-3ILY) currently has only one transport: `charmbracelet/huh` in the CLI via `rela flow <script.lua>`. The `lua.Transport` interface was deliberately designed to be transport-agnostic, and FEAT-YIOF explicitly lists \"Web transport (HTMX-based, integrate with data-entry app)\" as a future extension.\n\nUsers who run the data-entry web UI cannot trigger flows from the browser. This blocks real workflows (wizards, guided data entry, multi-step entity creation) for non-terminal users.\n\n## Goal\n\nImplement an HTTP/HTMX transport for the Lua flow system so that flows can be launched and completed entirely in the data-entry web UI, with each yielded screen rendered as an HTMX-driven form page.\n\n## Scope\n\n**In scope:**\n\n- New `lua.Transport` implementation backed by an HTTP request/response channel (the coroutine runs on the server; screens are rendered as HTML; form POSTs resume the coroutine).\n- Server-side flow session manager: keeps live coroutines in memory keyed by session ID, with TTL eviction.\n- HTML templates for rendering `Screen` types (form, markdown) and all existing field types (text, select, multi-select, boolean, number, date, markdown).\n- Dedicated `/flows` page in data-entry listing available flow scripts from the project's Lua script directory.\n- Ability for users to reference flows in `data-entry.yaml` `navigation` entries so they appear in the sidebar menu.\n- End-to-end tests covering: single-step form, multi-step wizard, cancellation, session expiry.\n\n**Out of scope:**\n\n- MCP transport (separate future ticket).\n- Persisting in-flight flow state across server restarts (server-side session is in-memory only).\n- New flow field types beyond what CLI already supports.\n- Changes to the Lua API itself (`rela.flow.emit` stays identical).\n\n## Acceptance Criteria\n\n1. A flow script that works with `rela flow foo.lua` in the CLI also works in the web UI without modification.\n2. Users can launch a flow from a dedicated `/flows` page in data-entry.\n3. Users can add flows to `data-entry.yaml` navigation so they appear in the sidebar menu.\n4. Multi-step flows correctly preserve state between steps via a server-side session.\n5. Cancelling a flow mid-step cleanly unwinds the coroutine (no goroutine leak) and returns the user to the launcher.\n6. Session TTL evicts abandoned flows; the UI shows a friendly \"session expired\" message on resume attempts.\n7. E2E tests cover the happy path and cancellation for a representative multi-step flow.
