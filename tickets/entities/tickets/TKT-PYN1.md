## Problem\n\nThe data-entry `App` uses `App.mu` (sync.RWMutex) to protect a bundle of reloadable fields: `Cfg`, `meta`, `g`, `styleMap`, `styledTypes`, `userDefaults`, `palette`, `userPalette`, `openAPIGen`. Every HTTP handler takes `RLock` via `reloadLockMiddleware`; reloads take `Lock`; mutation handlers perform a fragile lock-upgrade dance. This bundles two unrelated concerns (reload coherency and write serialization) into a single lock.\n\nSee `.ignored/locking-alternatives.md` §2.1 for the full analysis. The fix is to introduce an immutable `AppState` struct held via `atomic.Pointer`, so handlers capture a coherent snapshot without any lock.\n\nThis ticket combines the alternatives-doc Stages 1 and 2: introduce `AppState` with dual-write, then migrate handlers to `state.Load()`. Ship as two PRs under this ticket if that's cleaner.\n\n## Depends On\n\n- TKT-252Y (Workspace.mu refactor) — should land first to validate the atomic.Pointer pattern on smaller surface.\n\n## Scope\n\n**In scope:**\n\n- Define `AppState` struct containing the reloadable fields.\n- Add `state atomic.Pointer[AppState]` to `App`.\n- On construction and on `onReload`, build a fresh `AppState` and `Store()` it (dual-write: keep updating the old fields too so nothing breaks yet).\n- Migrate every handler to read via `state.Load()` at the top of the handler.\n- Migrate non-handler callers (e.g., `StartGitFetch`) to use `state.Load()` where appropriate.\n- Tests still work under `-race`.\n\n**Out of scope:**\n\n- Deleting `App.mu` (ticket T4).\n- Removing `reloadLockMiddleware` (ticket T4).\n- Introducing `writeMu` (ticket T3).\n- Removing the lock-upgrade dance (ticket T3).\n- Any behavior change.\n\n## Acceptance Criteria\n\n1. `AppState` struct exists and is populated on startup and every reload.\n2. Every HTTP handler reads config/meta/graph/style state via `state.Load()` once at entry, not via `a.Cfg` / `a.meta` / `a.g`.\n3. `App.mu` still exists and still does what it used to do (no deletions yet).\n4. All tests pass under `-race`.\n5. No behavior change observable from outside.\n\n## Risk\n\nDual-write is briefly awkward (two sources of truth during the migration) but necessary for an atomic ship. Tests must cover both the handler-side read pattern and the reload-side publish pattern.
