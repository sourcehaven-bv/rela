## Problem\n\n`repository.Repository.Sync(meta, g)` currently takes a mutable `*graph.Graph` and calls `g.Clear()` then repopulates it in place. This is the root cause of the graph-torn-during-reload race called out in RR-PA4Y and RR-FJH7: readers that hold a snapshot of `workspaceState` observe the graph reference in their snapshot going transiently empty during a reload.\n\nBy changing `repo.Sync` to **return a fresh `*graph.Graph`** instead of mutating one, `Workspace.Reload` can build the new graph alongside the new metamodel/search-index and publish all four fields atomically as a new `workspaceState`. Readers holding a pre-reload snapshot keep the pre-reload graph reference forever (guaranteed valid, never touched); readers loading after the publish see the post-reload graph. No transient empty state is ever observable.\n\n## Why this is split out of TKT-PNPI\n\nTKT-PNPI's scope expanded to introduce the `Tx` primitive, which is a ~1000-line change touching many callers. The `repo.Sync` signature change is standalone, small (~150 lines), and valuable on its own — it closes the graph-torn race without requiring the larger transactional refactor. Splitting it out gives us a shippable small PR and reduces TKT-PNPI to just the Tx work.\n\n## Scope\n\n**In scope:**\n\n- Change `Repository.Sync` signature from `(meta, g) (*SyncResult, error)` to `(meta) (*graph.Graph, *SyncResult, error)`. Internally: `g := graph.New()`, populate it, return it.\n- Update `Workspace.Reload` to receive the fresh graph and include it in the published `workspaceState`.\n- Update `Workspace.Sync` (the public wrapper) — it now needs to decide whether to publish the new graph or keep the current one. Since `Sync` is called from the CLI `rela sync` command and the test suite, and it's semantically "rebuild the in-memory graph from disk," it should publish the fresh graph too. This is a behavior improvement: today a failed `Sync` leaves the graph empty (because `g.Clear` ran before the error); tomorrow a failed `Sync` leaves the old graph in place.\n- Update callers of `repo.Sync`: `workspace.New` (initial construction), `workspace.Reload`, `workspace.Sync`, the CLI `sync` command, `workspace_test.go`, `repository_test.go`.\n- Delete `graph.Graph.Clear()` and `graph.Graph.rebuildAdjacency()` if no other callers remain.\n- Update the `Workspace` struct doc comment to remove the \"Known limitation: graph-torn-during-reload\" section.\n- Extend `TestConcurrentReloadStateSnapshot` to iterate the graph in reader goroutines (via `CheckCardinality` or `ValidateProperties`). It should now pass under `-race` without the reader restrictions that TKT-252Y had to add.\n- Resolve RR-PA4Y and RR-FJH7 as addressed (not deferred).\n\n**Out of scope:**\n\n- The `Tx` primitive itself — that stays in TKT-PNPI.\n- Automation staging on the tx — stays in TKT-CXHM.\n- Clone-and-publish commit semantics — stays in TKT-1LFQ.\n- Any change to `Workspace.Graph()` callers. The accessor still returns the current graph from the state snapshot.\n\n## Acceptance Criteria\n\n1. `repository.Repository.Sync` returns a fresh `*graph.Graph`. `g.Clear()` calls are removed from `Sync`.\n2. `Workspace.Reload` publishes the fresh graph as part of the new `workspaceState`. The previous graph reference remains valid (never mutated) for readers holding pre-reload snapshots.\n3. `Workspace.Sync` also publishes the fresh graph. On failure, the old graph stays live.\n4. `graph.Graph.Clear()` is deleted if no other callers remain.\n5. `TestConcurrentReloadStateSnapshot` is extended to iterate the graph in readers and passes under `-race`. The \"Known limitation: graph-torn-during-reload\" section is removed from the `Workspace` struct doc comment.\n6. RR-PA4Y and RR-FJH7 move from `deferred` to `addressed`.\n7. `go test -race ./...`, `just lint`, `just coverage-check` all pass.\n\n## Dependencies\n\n- TKT-252Y (PR #342, pending merge): this ticket modifies the `workspaceState` struct introduced there, so it depends on that PR being in place.
