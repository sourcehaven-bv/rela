## Summary

Add a PostgreSQL-backed implementation of `store.Store` and use PostgreSQL for
the search index too (no bleve in that build). Ship FS and PostgreSQL variants
of `rela-server` and `rela` (CLI) selected by Go build tags, and update CI /
release so both variants are built and released.

## Motivation

Today the only persistent store is `fsstore` (markdown files + in-memory bleve
search). A PostgreSQL backend enables multi-writer / server deployments,
concurrent access, and pushing search into the database (full-text via
`tsvector`/`pg_trgm`) instead of an in-process bleve index.

The codebase is already architected for this: `internal/appbuild` and
`internal/cli` use build-tag companion files (`!postgres && !memorybackend`,
`memorybackend`), and the existing FS/memory files carry comments explicitly
anticipating a `//go:build postgres` companion. `FEAT-CO4YP` ("Pluggable store
backends with conformance test-kit") and the `storetest` conformance harness
provide the contract a new backend must satisfy.

## Scope

In scope:

- `internal/store/pgstore`: a `store.Store` implementation backed by PostgreSQL
(entities, relations, attachments, watcher events, lifecycle, freshness),
passing the full `storetest.RunAll` conformance suite (incl. fuzz +
attachments).
- PostgreSQL-backed search: a `search.Backend` (or equivalent) that indexes
inside the database (tsvector / pg_trgm), so the `postgres` build does NOT link
bleve. Wire via the existing `newSearchObserver`/`buildSearcher` seam.
- `//go:build postgres` companion files for the existing seams:
`appbuild/store_postgres.go`, `appbuild/search_postgres.go`,
`cli/mcp_wiring_store_postgres.go`, `cli/mcp_wiring_search_postgres.go`.
- Connection configuration (DSN/env) for `rela-server` and `rela` in the
postgres build, plus schema migrations bundled with the binary.
- CI/release: build & release both FS (default) and PostgreSQL variants of
`rela` and `rela-server` (goreleaser + release.yml). Add a CI job that runs the
pgstore conformance suite against a real PostgreSQL service container.

Out of scope (this ticket):

- `rela-desktop` PostgreSQL variant (desktop stays FS).
- Migrating existing FS projects into PostgreSQL (data migration tooling) —
separate ticket if needed.
- Connection pooling tuning / HA / replication topology.

## Acceptance criteria

- `go build -tags postgres ./cmd/rela ./cmd/rela-server` produces working
binaries that talk to PostgreSQL; bleve is NOT in the dependency graph of the
postgres build (verify with `go list -deps -tags postgres`).
- `pgstore` passes `storetest.RunAll(...)` including the search factory and
attachment capability, with the race detector on, against a real PostgreSQL.
- FS build is byte-for-byte behaviorally unchanged (default build still uses
fsstore + bleve).
- CI builds both variants; release publishes both variants of `rela` and
`rela-server`.
- A new CI job spins up PostgreSQL (service container) and runs the pgstore
conformance + integration tests.

## Open questions (resolve in planning)

- Driver: `pgx` (pgx/v5 stdlib vs native) vs `lib/pq` — likely pgx/v5.
- Migration tooling: embedded SQL via `golang-migrate`/`goose`/hand-rolled
`embed.FS` + version table.
- Full-text strategy: `tsvector` GIN vs `pg_trgm` — match bleve's fuzzy/wildcard
semantics (`FEAT-y6g5`) as closely as practical.
- Watcher/event delivery: PostgreSQL `LISTEN/NOTIFY` vs polling for the
`store.Watcher` subscription contract.
- How DSN is supplied for the CLI (env `RELA_DATABASE_URL`? flag?).
