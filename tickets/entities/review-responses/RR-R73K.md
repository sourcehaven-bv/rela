title: user-overrides-project rule for Dark is asymmetric — user can't reset to project default
finding: "internal/dataentryconfig/palette.go:271-280: the user palette wins for Dark only if the user has `Disabled=true` OR a non-nil `Explicit`. This means a user CAN'T express 'I want dark to fall back to whatever the project says' once they've overridden it — they have to delete their palette.yaml entirely (because the new in-UI 'Reset' button just reloads from disk). \n\nFlip side: if the project ships `dark: { accent: '#abc' }` and the user is in Regular mode, the user palette has zero-value DarkMode (neither Disabled nor Explicit), so the project's dark wins → the resolved palette has DarkDisabled=false even though the user explicitly chose Regular mode. Then `buildPalettePayload` sends `dark: false` over the wire, the backend stores `dark: false` in the user file, and the user sees Regular mode in the UI but the *resolved* palette still has dark colors from the project — wait, no, `dark: false` has Disabled=true which DOES override the project. OK, that path is fine.\n\nBut: a user who never visits Settings has `userPalette == nil`, and `dark: { accent: '#abc' }` from the project propagates to their browser's resolved palette. As soon as they open Settings, `loadPaletteState` sees `data.userPalette` (which is the *user* palette only — `data.userPalette` does not contain merged project settings), so the UI initialises in Regular mode regardless of what the project shipped. First click of Save Palette flips the project's dark off for that user. That's data loss too — the user never saw the project's dark theme as 'theirs' but their action of saving silently disabled it.\n\nFix: the Settings UI needs to load the *resolved* palette to seed the editor (not just the user-only overlay), AND probably needs a tri-state per slot ('inherit / set to X'). Short-term: at least display 'Project provides a dark theme' as a hint in the Mode section so the user knows what they're about to lose by saving Regular."
severity: significant
resolution: 'Added third parameter `resolvedDarkDisabled` to `loadPaletteState` in `SettingsView.palette.ts`. When the user palette has no explicit `dark` field (neither false nor object), the helper now consults the resolved palette state (passed in from `schemaStore.darkDisabled`). If the project ships a dark theme, the editor opens in Light+Dark mode so a naive Save doesn''t shadow the project''s dark with `dark: false`. SettingsView.vue now passes `schemaStore.darkDisabled` when loading. Added 4 new Vitest tests in `SettingsView.palette.test.ts` covering all four combinations of (user dark explicit/missing) × (project dark enabled/disabled).'
status: addressed
