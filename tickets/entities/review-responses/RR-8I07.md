id: RR-8I07
type: review-response
title: ValidateCreate scans the entire entity store per keystroke (generateID)
finding: 'buildCandidateEntity calls generateID for auto-ID types when opts.ID is empty, which calls collectAllIDs -> store.ListEntities({}) over the entire store. The dry-run runs on every debounced keystroke (400ms) in the create form, so on a project with N entities the dry-run does an N-row scan per edit. On a 10k-entity project that''s a noticeable hitch and unnecessary load. The dry-run doesn''t need a real ID at all - only the validation. Fix: in ValidateCreate, when opts.ID is empty for an auto-ID type, synthesize a placeholder (e.g. ''__candidate__'') and skip generateID. ValidateID still needs to pass; or thread a ''skipIDGen'' flag through buildCandidateEntity. Manual-ID branch keeps its real validation.'
severity: significant
resolution: Added SkipIDGeneration to createCoreOpts; Manager.ValidateCreate sets it true so buildCandidateEntity uses candidatePlaceholderID (prefix+'DRYRUN') instead of scanning the store. ID-prefix validation still passes (the placeholder is built from the type's first declared prefix). createCore (persisting path) unchanged. New TestValidateCreate_SkipsIDGeneration uses a countingStore to assert ListEntities is NOT called during ValidateCreate.
status: addressed
