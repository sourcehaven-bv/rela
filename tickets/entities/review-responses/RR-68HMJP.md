title: _apps/{id} placement under /api/ breaks direct navigation; clarify it's srcdoc-only and document origin-null as the backstop
finding: 'requireSameOrigin applies to ALL methods including GET (middleware_security.go:141-163, by design per comment :24-27) and treats everything under /api/ as sensitive (:197-208). Two consequences the plan should address explicitly: (1) A top-level browser navigation directly to /api/v1/_apps/{id} sends no Origin on a same-origin GET nav → requestOrigin returns ('''',false) → origin_missing → 403. So the app HTML cannot be opened by direct navigation when under /api/. That''s FINE for this design (apps are only ever loaded via the host fetching into srcdoc), but the plan currently also says ''serve with CSP so direct navigation is safe'' — direct navigation isn''t even reachable, so that rationale is moot and should be removed/clarified. (2) The sandboxed iframe is origin ''null'' (allow-scripts WITHOUT allow-same-origin); requestOrigin rejects ''null'' (:216-218) → the iframe physically CANNOT call /api/ directly → 403. This is a SECURITY FEATURE: it forces all API traffic through the host MessageChannel bridge and is the backstop that makes the closed-allow-list meaningful. The plan should (a) document origin-null blocking as a deliberate backstop, (b) assert allow-same-origin is never set, (c) confirm the host bridge is the SOLE API path. Net: keep _apps/{id} under /api/ (origin-gated, good), serve as srcdoc source only, and document the two points above. Confirmed: middleware_security.go:141-234.'
severity: minor
resolution: 'Resolved in the plan. _apps/{id} stays under /api/ (origin-gated). Plan corrected to: (a) remove the ''serve with CSP so direct navigation is safe'' rationale — direct nav to an /api/ path 403s on origin_missing anyway and is never used; apps load ONLY via the host fetching the HTML into srcdoc; (b) document origin-null blocking (requestOrigin rejects ''null'', middleware_security.go:216-218) as a deliberate SECURITY BACKSTOP that forces all API traffic through the host MessageChannel bridge; (c) assert allow-same-origin is NEVER set on the iframe; (d) confirm the host bridge is the SOLE API path for apps. No middleware change needed (/api/ prefix already covers _apps/).'
