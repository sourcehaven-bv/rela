title: attachACLRequest silently continues on unstamped principal — read-side becomes fail-open
finding: 'Plan''s Security Considerations says ''attachACLRequest panics on unstamped principal when ACL is configured, so this can''t reach the read handlers.'' Reading router.go:125-138: the middleware actually does `req, err := d.ForPrincipal(...); if err == nil { ctx = acl.WithRequest(ctx, req) }; next.ServeHTTP(...)`. On ForPrincipal error (unstamped principal — ErrUnstampedPrincipal), it silently continues with no Request attached. The plan''s read-side conditional `if req == nil { /* today''s path */ }` therefore means: with ACL configured + a misconfigured principal stamper, every read becomes AllowAll. This is a fail-open regression introduced by this PR. Fix (preferred): make attachACLRequest fail loud (500 or panic) when ACL is configured AND the principal is unstamped — matches the godoc on ErrUnstampedPrincipal and rela house style. Alternative: change the read-side conditional to ''no Request AND no ACL configured -> AllowAll; no Request AND ACL configured -> DenyAll''. Audit the middleware before pinning the invariant in tests.'
severity: critical
resolution: 'Incorporated into rescoped TKT-VQGN scope: attachACLRequest returns 500 + structured error when ACL configured AND ForPrincipal errors. Pinned by AC7. (500 not panic — per-request misconfig shouldn''t take down the process.)'
status: addressed
