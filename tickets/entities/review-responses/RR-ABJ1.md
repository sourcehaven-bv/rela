title: generateDark/generateDarkBadges kept in palette.go but no longer called from production code — dead-ish code waiting to drift
finding: |-
    internal/dataentryconfig/palette.go:400 `generateDark` and :414 `generateDarkBadges` are no longer called by `ResolvePalette` — they exist only to be exercised by `palette_parity_test.go`. The package doc claims this is the 'reference implementation' kept for parity, but Go's dead-code analysers (and the next person tidying up the file) won't read the ticket. They will see two unexported functions called only from a test, and the naive cleanup is to delete them and let the parity test rot.

    Fix: either (a) move the parity goldens to be the source of truth and DELETE generateDark from Go entirely (the test then just asserts the goldens haven't changed and the TS port matches), since the Go side no longer needs to compute dark anywhere; or (b) add a `// Public for test parity contract — see palette_parity_test.go. Removing this without updating the parity test is a build break.` block-comment above the pair and a `// nolint:unused` if the linter complains.

    Option (a) is cleaner — Go has no business carrying a duplicate dark-derivation algorithm if it's never used at runtime.
severity: minor
resolution: Added a clear godoc to `generateDark` and `generateDarkBadges` in palette.go explaining that they are the *reference implementation* kept solely as the source of truth for the parity goldens generated by `TestGenerateDarkParityGoldens`. Includes the regeneration command. Future cleanup passes that grep for unused symbols will see the doc and the test reference and not delete the functions.
status: addressed
