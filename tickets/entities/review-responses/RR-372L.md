title: attachACLRequest leaks raw err.Error() in the 500 body
finding: 'writeV1Error(w, r, 500, ''acl_unstamped_principal'', ''...'', err.Error()) puts the full error in the Detail field of application/problem+json. Today''s wrap is harmless (`User=unknown Tool=unknown`), but ForPrincipal''s contract leaves room for future enhancements (LDAP errors, config-file paths, internal user identifiers). The middleware runs on every /api/ request including unauthenticated probes, so a future ForPrincipal enhancement that surfaces internal details silently becomes attacker-readable. Also there''s no server-side log line, no audit event, no metric on this path — operator gets nothing in logs. Fix: log err server-side with path/method/principal-as-reported; pass a constant attacker-safe string in the Detail field.'
severity: critical
resolution: attachACLRequest logs the raw err via slog.Warn with path/method/principal; the response body now carries a constant attacker-safe Detail string.
status: addressed
