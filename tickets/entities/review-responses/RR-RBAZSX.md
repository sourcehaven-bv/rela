title: OpRunApp as a WriteRequest breaks under ReadOnlyACL and doesn't fit the policy schema
finding: 'Modeling ''run app'' as acl.WriteRequest{Op:OpRunApp, Subject:AppSubject} collides with three hard facts in internal/acl/: (1) ReadOnlyACL.AuthorizeWrite (readonly.go:21) unconditionally returns Allow:false for EVERY WriteRequest, ignoring Op/Subject — so under --read-only mode a read-only app becomes unrunnable, which is wrong (running an app is conceptually a READ; reads still work in read-only mode via the separate readGate/acl.Request.PermitsRead path). (2) acl.Subject is a SEALED interface (subject.go:20, unexported isSubject()) and authorizeWrite''s type switch PANICS on an unhandled variant (authz_write.go:26-30) — an AppSubject needs an in-package type + a switch arm or it panics, not denies. (3) grantsVerb (policy.go:108-117) returns false for unknown Ops by default and grants are strictly per-entity-type verb lists (RoleDef.Create/Update/Delete/Read); there is no app/coarse-grant slot, so OpRunApp through grantsVerb denies everything unless the policy schema is extended. The ACL interface has NO read Op at all — reads go through acl.Request.PermitsRead / readGate, an Op-less mechanism that already permits reads under ReadOnlyACL. FIX: model ''run app'' as a READ-style decision on the read path (extend readGate / acl.Request with a PermitsRunApp-style method, or gate the _apps handler via the existing read gate using a coarse capability), NOT as a write Op. This avoids the panic, the read-only breakage, and the schema mismatch. Reconsider whether a dedicated ACL change is even needed for v1, or whether the existing read gate + config-presence is sufficient.'
severity: significant
resolution: 'Resolved by DROPPING the ACL change entirely. Decision: an app is just a UI shell; every read/write it performs goes through the host page''s existing per-call read gate (acl.Request/readGate) and write authz+audit (entitymanager). Therefore an app can only ever do what the logged-in user can already do — no dedicated ''run-app'' authorization is needed. No OpRunApp, no AppSubject, no policy-schema change, no ReadOnlyACL/panic/schema-mismatch problems. v1 gates app availability on config presence (app must be declared in data-entry.yaml) only. Accepted caveat: any authenticated SPA user can load any declared app''s inert HTML/JS source (project config, authored via git/PR, not user data); a per-app/role gate can be added later if needed. Plan updated to remove all internal/acl changes.'
status: addressed
