title: Hidden field with a non-null default cannot be staged client-side correctly
finding: 'Plan says create seeds the staged entity from metamodel + form defaults, then derives _fields. But if a field is HIDDEN for this user yet has a metamodel default value, the staged entity would carry that default in its properties. Two problems: (1) the form must omit the hidden field from the UI but still send its defaulted value at commit (or not?) — unspecified; (2) the initial mount derivation is circular: you need _fields to know what''s hidden, but you seed properties (including would-be-hidden defaults) before the first derivation returns. Clarify: does a hidden field''s default get committed silently, dropped, or is hidden-with-default a policy authors must avoid? The server gate (BUG-Q60V) currently 403s a SET of a hidden field — so committing a hidden field''s default would 403 the whole create. This is a real deadlock: a hidden field with a required default makes the type uncreatable via the form.'
severity: significant
resolution: 'Decision: server applies defaults post-gate (option 1). Verified this already works in the existing create path: handleV1CreateEntity runs the field-affordance gate against req.Properties (client keys only), THEN entitymanager.CreateEntity (core.go:66-80) applies template + metamodel + status defaults, merging client properties on top (lines 70-72). So a hidden field the form omits gets its default filled server-side, downstream of the gate, never passing through it as a ''user write''. Resolution for the plan: (a) the SPA create form must NOT send hidden/read-only fields in the commit body (only visible-and-writable keys it actually rendered); (b) no new server code needed for default application — the template path already covers it; (c) mount ordering wrinkle dissolves: the client seeds only visible fields'' defaults after the first derivation, hidden defaults are purely server-side, no circular dependency. Plan updated.'
status: addressed
