id: RR-PA4Y
type: review-response
title: Reload leaves graph torn during repo.Sync; no serialization
finding: The old code used RWMutex.Lock() for Reload(), which excluded all readers during repo.Sync(). The new code has zero synchronization. repo.Sync calls g.Clear() on repository.go:268 and repopulates the graph. During that window, readers see an empty graph with the old metamodel and old search index. Search returns hits from the old index, but graph.GetNode filters them all out (false-empty). Analyze tools return false negatives. Two concurrent Reload() calls (watcher + user) can interleave Clear+repopulate sequences with undefined outcome. The struct comment says readers never see torn values, but the graph is not an atomic.Pointer — it is mutated in place.
severity: critical
resolution: 'Fixed in TKT-Z7HL: repo.Sync now returns a fresh *graph.Graph instead of mutating one in place. Workspace.Reload publishes a new workspaceState containing {graph, meta, automation, searchIdx} via a single atomic.Pointer.Store. Readers holding a pre-reload state continue to see a fully-populated graph (never mutated by Reload). The graph field has been folded into workspaceState so meta+automation+searchIdx+graph are always observed as a coherent tuple from a single Load. Workspace.Sync also publishes atomically via the same path. graph.Clear() has been deleted from internal/graph/graph.go since no callers remain. The known-limitation section is removed from the Workspace struct doc.'
reason: 'Deferred from TKT-252Y because the fix requires changes that are out of scope for a Workspace.mu replacement: repo.Sync needs to return a fresh graph instead of mutating one in place, the Workspace.graph field needs to move into workspaceState so a single atomic.Pointer publish covers all reloadable fields coherently, and several non-trivial callers (workspace.New, workspace.Reload, the data-entry watcher tests, repository tests, and the CLI sync command) need to migrate to the new repo.Sync signature. The narrow TKT-252Y refactor only swapped Workspace.mu for atomic.Pointer; doing the graph swap in the same PR would have ballooned scope and risked merge conflicts with the writer-locking work tracked in TKT-PYN1, TKT-WYYP, and TKT-9NFK. The fix is tracked as a follow-up ticket TKT-PNPI within the same FEAT-W5T8 refactor program. Note that this bug is genuinely pre-existing rather than introduced by TKT-252Y: the old RWMutex-based code released its RLock in Search before iterating the graph, so a concurrent Reload running g.Clear could already produce false-empty results.'
status: addressed
