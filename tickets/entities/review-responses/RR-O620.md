title: 'Validation policy for inserted ID strings: allowlist vs. blocklist'
finding: 'Plan describes `insertEntityRef` rejecting IDs that contain `\``, newlines, or whitespace — a blocklist. This is the classic security smell. The right shape is an allowlist: assert the ID matches the metamodel''s ID pattern (`[A-Z]+-[A-Za-z0-9]+` for short IDs, or the freeform manual-ID shape). The picker only ever receives IDs returned by the server''s `/_search`, so in practice the source is trusted — but the helper should still allowlist-validate so a future caller (e.g. a Lua-driven action) doesn''t get a free injection vector. The validation also needs a max length cap (e.g. 256 chars) to prevent an absurdly long ID from blowing up the editor.'
severity: significant
resolution: 'Plan §Approach §3 + Security: switched to allowlist /^[A-Za-z][A-Za-z0-9_-]{0,255}$/ covering both short and manual IDs, max 256 chars. AC 10 plus negative tests for empty/backtick/whitespace/newline/overlong inputs.'
status: addressed
