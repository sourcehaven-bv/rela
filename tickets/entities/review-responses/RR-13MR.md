id: RR-13MR
type: review-response
title: Empty PropertyOrder defeats every inaccessible guard — critical data-loss path
finding: 'internal/store/fsstore/markdown.go:370-384 buildInaccessibleEntity calls s.propertyOrder(entityType). When the entity type has no Schemas entry or empty PropertyOrder, props is nil/empty, Inaccessible becomes []InaccessibleField{} (length 0). Every downstream guard checks `len(e.Inaccessible) > 0` (validator.go:170, api_v1.go:527). Result: a PATCH against an encrypted entity whose type lacks PropertyOrder bypasses the 422 reject and overwrites ciphertext with cleartext form data — the exact data-loss the ticket exists to prevent. Fix: when props is empty, fall back to a single sentinel `{Name: "*", Reason: gitCrypt}` so the guards still trigger; OR change all consumers to check via a helper like `e.IsLocked()` that knows about the wildcard.'
severity: critical
resolution: 'Closed by enforcement, not defensive coding. Three changes: (1) fsstore.New now rejects empty Schemas with a clear error — catches bootstrap-ordering bugs at startup. (2) resolveEntityType returns empty for unknown plurals, and scanEntityDirs / entityIdentityFromPath skip empty results — directories whose plural does not map to a metamodel-declared type are invisible to the loader, so buildInaccessibleEntity is never called for an unknown type. (3) Added Entity.IsLocked() / Relation.IsLocked() helpers; replaced every `len(.Inaccessible) > 0` site with the helper. Plus paranoid fallback: when PropertyOrder is empty (legitimate property-less entity type), buildInaccessibleEntity emits a {Name: ''*''} sentinel so IsLocked() still triggers. New regression tests: TestNew_RejectsEmptySchemas, TestScan_SkipsUnknownEntityTypeDirectories, TestGitCrypt_PropertylessEntityType_StillLocks. The bug class is now structurally impossible: an encrypted file either is rejected at scan (unknown type) or produces an entity with IsLocked()==true.'
status: addressed
