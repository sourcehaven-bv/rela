description: 'Adds a Processor seam to the attachment write path (Service.WriteAttachment, the single chokepoint shared by CLI and HTTP upload) so uploads can be inspected and transformed before storage. Processing is delivered by ONE generic cmd: external-command processor: rela ships no native scanners or transformers (no clamd client, no hand-rolled EXIF stripper, no native resize) — virus scan, metadata strip, resize, and CDR are all driven by operator-configured external commands (clamav, vips, exiftool, qpdf, Defender, ffmpeg, DLP) with vetted recipes shipped in the docs. This mirrors rela''s existing house style of outsourcing auth to the proxy: push integration to a well-understood external boundary, keep the core small. The cmd: mechanism enforces safe invocation (array args never a shell string, templated {in}/{out} paths owned by rela, timeout, output-size cap, present-probe + startup warn). Input VALIDATION that needs no external tool stays native and default-on: a default-safe MIME allowlist validated against the sniffed type (rejecting SVG/HTML/executables and sniff-vs-extension mismatches) plus download hardening headers (Content-Disposition: attachment, X-Content-Type-Options: nosniff). Since scan/strip are opt-in via recipe, an unset-scan startup warning nudges operators to make a conscious choice (config distinguishes unset from explicit off). Lua serves only as an optional conditional-policy layer selecting/gating cmd: processors. Phased 0 (seam + conditional buffering) -> 1 (native MIME validation + download hardening + unset-scan warning) -> 2 (cmd: processor harness + doc recipes + scan/transform config); Phase 3 (Lua policy) deferred. Grounded in RES-INM8JP.'
