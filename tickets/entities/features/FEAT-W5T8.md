## Summary\n\nReplace the RWMutex-based state protection in the data-entry server with `atomic.Pointer`-based snapshots. Split the two concerns currently bundled in `App.mu` (reload coherency + write serialization) into two separate primitives: an atomic `AppState` pointer for reads, and a dedicated `writeMu` for mutation serialization.\n\n## Motivation\n\nThe current locking scheme bundles two unrelated responsibilities in one `App.mu sync.RWMutex`:\n\n1. **Reload coherency** — keep `a.Cfg`, `a.meta`, `a.g` consistent for the duration of a request against concurrent file-watcher reloads.\n2. **Write serialization** — prevent two mutation handlers from racing on the workspace.\n\nThis bundling is the root cause of three concrete pains:\n\n- The lock-upgrade dance in every mutation handler (`RUnlock → Lock → defer(Unlock → RLock)`), which is subtle and error-prone.\n- The \"30s under global write lock is a DoS\" problem already identified in RR-CSTD, mitigated only by tightening the action timeout to 5s.\n- It makes interactive features (like the web transport for Lua flows in FEAT-YIOF / TKT-011J) impossible without either holding the server hostage during user think time or re-architecting first.\n\n## Approach\n\nSee `.ignored/locking-alternatives.md` for the full analysis. Summary:\n\n1. Reloadable state (`Cfg`, `meta`, `g`, `styleMap`, etc.) moves into an immutable `AppState` struct, held via `atomic.Pointer[AppState]`. Handlers snapshot it once at entry; reloads publish a new pointer.\n2. `Workspace.searchIdx` and `Workspace.meta` likewise move to `atomic.Pointer`.\n3. Mutations serialize via a new `writeMu sync.Mutex`, independent of reload coherency.\n4. Once handlers no longer read through `App.mu`, the RWMutex and `reloadLockMiddleware` can be deleted entirely.\n\n## Scope\n\n**In scope:**\n\n- `Workspace.mu` → `atomic.Pointer`\n- `App.mu` → `atomic.Pointer[AppState]`\n- New `writeMu sync.Mutex` for mutation serialization\n- Deletion of `reloadLockMiddleware`\n- Deletion of the lock-upgrade dance in every mutation handler\n\n**Out of scope (future work):**\n\n- Narrowing the script-execution contract (multi-op atomicity → per-op + opt-in transaction). This is Stages 5-6 in the alternatives doc; it's a behavior change and deserves its own feature.\n- Any change to `Graph.mu`. The alternatives doc explicitly argues to keep it.\n- The web transport for Lua flows (TKT-011J). This feature unblocks that ticket but does not include it.\n\n## Acceptance Criteria\n\n1. `App.mu` is deleted.\n2. `Workspace.mu` is deleted (or reduced to a no-op).\n3. `reloadLockMiddleware` is deleted.\n4. Mutation handlers use `writeMu`; no lock-upgrade dance remains.\n5. All existing tests pass under `-race`.\n6. A new test covers concurrent reload-during-read without blocking.\n7. No behavior change observable from outside the server.\n\n## Related\n\n- FEAT-YIOF / TKT-011J — blocked by this refactor\n- RR-CSTD — prior review finding that motivated the 5s action timeout; this refactor removes the underlying cause\n- RR-Z3R9 — prior review finding about action handler workspace locking\n- `.ignored/locking-findings.md` — the investigation that preceded this refactor\n- `.ignored/locking-alternatives.md` — the design rationale
